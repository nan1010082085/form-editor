# 可视化编辑器 · 产品迭代进化路线

> 视角：从产品成熟度与用户价值出发，规划 Editor 从「表单设计器」到「页面/大屏搭建器」再到「开放搭建平台」的演进路径。
> 本文承接 [editor-review-and-roadmap.md](./editor-review-and-roadmap.md) 的技术任务清单，聚焦产品维度的不足诊断、演进决策与验收度量。
> 日期：2026-07-20 · 作者：产品视角评审

## 一、现状诊断（产品视角）

### 1.1 产品定位的张力

当前 Editor 是「Schema 驱动的可视化表单设计器」，正向「页面/大屏搭建器」演进，但演进尚未闭环：

- 已有基础（可直接复用）：85+ widget、free/flex 双布局、17 种 ECharts 图表、全局数据源中心（DataSourceStore）、四大配置系统（事件/联动/API/变量）、AI page agent。
- 缺口（卡住产品天花板）：画布无虚拟化（widget > 100 卡顿）、无大屏主题/动画、4 种交互模式只用了 edit/preview 两种、Widget 类型硬编码在核心类型文件限制第三方扩展。
- 风险判断：表单与大屏两头兼顾，若不聚焦，可能两边都不极致。这是当前最大的产品决策风险。

### 1.2 核心不足清单（按产品影响排序）

| 编号 | 不足 | 产品影响 | 根因 | 归属阶段 |
|------|------|----------|------|----------|
| E-01 | 文档与代码统计严重脱节 | 新人上手被误导，AI 协作基于错误数字 | README/CLAUDE.md/architecture.md 三处对 Store(7/11/12)、Widget(49/64/~85)、Composable(32/37/46) 互相矛盾 | E1 立即修 |
| E-02 | 画布无虚拟化 | 大屏场景 > 100 widget 卡顿，卡死「大屏搭建器」定位 | free 布局全量渲染，ComponentPanel 的 virtualWrapper 仅是 CSS 类名 | E1 |
| E-03 | 撤销/重做 JSON 深拷贝 | 大 schema 下「画一会儿就卡」，丢失函数/Date 类型 | `JSON.parse(JSON.stringify())` 全量快照，30 步上限 | E1 |
| E-04 | Widget 类型硬编码 | 第三方无法零侵入扩展，与「插件式注册」目标矛盾 | 约 80 个类型字面量写在 base/types.ts，新增需改核心类型文件 | E3 |
| E-05 | i18n 框架空转 | 实质是中文单语言产品，阻碍出海与开源 | useLocale.ts + 两套语言包就绪，但语言包仅约 20 个 key | E2 |
| E-06 | 4 种模式只用 2 种 | 超前设计未落地，发布态交互能力缺失 | INTERACTION_MODES 定义了 publish-interactive/publish-readonly 但未实现 | E2 |
| E-07 | 无生产监控/埋点 | 无法度量用户流失点、高频操作、性能瓶颈 | 全局错误仅 console.error，无 Sentry/自研上报 | E2 |
| E-08 | deprecated 标记未清理 | 技术债累积，误导使用者 | WidgetRule（应改用 SchemaLinkage）、getAllContainerTypes/getTypesByGroup | E1 |
| E-09 | PropertyPanel.vue 37KB | 属性面板维护成本高，新属性接入慢 | 单文件集中所有属性编辑器分发 | E2 |
| E-10 | 容器不能嵌套容器 | 限制页面嵌套布局能力 | 自动提升到根级，未放开 | E2 评估 |

### 1.3 成熟度评估

| 维度 | 评分（5 分制） | 说明 |
|------|----------------|------|
| 架构设计 | 4.5 | 双路径渲染、纯逻辑事件引擎、四大配置系统抽象到位 |
| 工程化 | 4.0 | strict TS、55 个测试、注释充足，但文档统计漂移扣分 |
| 功能完整度 | 3.5 | 表单侧完整，页面/大屏侧地基有但未闭环 |
| 用户体验 | 3.0 | 撤销卡顿、无虚拟化、无埋点优化依据 |
| 开放性 | 2.5 | 注册机制有，但类型硬编码限制第三方扩展 |
| 可观测性 | 2.0 | 无生产监控，错误仅 console |

**总体判断**：架构与工程化是强项，体验与开放性是短板。当前是「技术领先于产品闭环」的典型状态——地基扎实，但用户感知到的「丝滑」与「开放」还没跟上。

## 二、演进目标

### 2.1 北极星

让非开发人员也能像搭积木一样，搭建从审批表单到运营大屏的任意可视化页面，并支持发布为可运行页面、嵌入微前端、对接审批流程。

### 2.2 三阶段目标

| 阶段 | 目标 | 衡量标准（可验收） |
|------|------|---------------------|
| E1 大屏地基 | 让页面/大屏模式「真正可用」 | 100 个 widget 下编辑流畅（FPS ≥ 30）；撤销重做不卡；大屏模板可发布 |
| E2 体验深化 | 让搭建「丝滑」、可度量 | i18n 覆盖 ≥ 80%；前端埋点上线；4 种模式全落地 |
| E3 开放生态 | 让第三方能扩展 | Widget 类型从硬编码改为完全注册式；首条第三方 widget 接入文档 |

## 三、分阶段进化路径

### Phase E1：补齐大屏地基（让页面模式真正可用）

**目标**：消除「画一会儿就卡」的体感，让 100+ widget 的大屏场景可编辑可发布。这是产品从「表单设计器」迈向「大屏搭建器」的最低门槛。

| 编号 | 关键工作 | 验收标准 | 依赖 |
|------|----------|----------|------|
| E1.1 | 文档与代码统计对齐 | README/CLAUDE.md/architecture.md 三处 Store/Widget/Composable 数量一致且与代码同步；建立「改代码必改文档」的 CONTRIBUTING 约束 | 无 |
| E1.2 | 画布虚拟化 | free 布局下 widget > 100 时只渲染可视区域 + 缓冲区；编辑 FPS ≥ 30；有专项性能测试（dragOptimization.spec.ts 扩展） | 无 |
| E1.3 | 撤销/重做增量 diff | 替换 JSON 深拷贝为增量 diff 方案；大 schema（> 500 widget）下 undo/redo 响应 < 100ms；不丢失函数/Date 类型 | 技术预研 |
| E1.4 | deprecated 标记清理 | 移除 WidgetRule、getAllContainerTypes、getTypesByGroup；确认无引用后删除 | 全量回归 |
| E1.5 | 大屏主题与动画 | 深色主题预设；widget 入场动画 + 数据变化动画（roadmap Phase 2.6） | 主题系统设计 |

**关键决策**：E1.2 画布虚拟化是本期最高风险项。需评估「绝对定位 + 虚拟窗口」方案对拖拽/辅助线/选中态的影响，可能需要重写 EditorOverlay 的命中计算。

**预期产出**：一份完整可发布的大屏 demo（10+ 图表、联动、实时刷新），作为 E1 验收的硬性指标。

### Phase E2：体验与质量深化（让搭建「丝滑」）

**目标**：从「能用」到「好用」。建立可观测性，让产品迭代有数据依据；补齐国际化与发布态交互。

| 编号 | 关键工作 | 验收标准 | 依赖 |
|------|----------|----------|------|
| E2.1 | 前端埋点与监控接入 | 关键路径埋点（拖入/删除/发布/模板使用/撤销频率）；错误上报（Sentry 或自研）；dashboard 可见用户流失点 | 平台侧上报通道 |
| E2.2 | i18n 实质推进 | 语言包从 ~20 key 扩展到覆盖全部 UI 文案（≥ 80%）；EditorView/PropertyPanel/Toolbar 全量接入 useLocale | 抽取文案为 key |
| E2.3 | 发布态模式落地 | publish-interactive / publish-readonly 两种模式实现；已发布页面支持交互与只读切换 | E1 完成 |
| E2.4 | PropertyPanel 拆分 | 按分组（basic/style/props/events/linkages）拆为子组件；单文件 < 10KB；属性接入成本降低 | 无 |
| E2.5 | 容器嵌套策略评估 | 决策是否放开「容器嵌套容器」；若放开需递归碰撞检测与 reparent 逻辑加固 | 性能测试 |
| E2.6 | 快捷键补齐 | 新增对齐（左/右/居中）、等距分布、成组/解组、锁定/隐藏；快捷键面板可查看 | 无 |

**关键决策**：E2.5 容器嵌套是产品能力与复杂度的权衡。放开可支持更复杂的页面布局，但碰撞检测与 reparent 逻辑复杂度指数上升。建议先做用户场景调研，确认「嵌套容器」是否为高频需求，再决定。

### Phase E3：开放生态（让第三方扩展）

**目标**：从「自建组件库」到「开放组件生态」。让第三方能零侵入扩展 widget，支撑「页面/大屏搭建器」的长期生命力。

| 编号 | 关键工作 | 验收标准 | 依赖 |
|------|----------|----------|------|
| E3.1 | Widget 类型注册式改造 | base/types.ts 的 BasicType/ContainerType/BusinessType 联合类型改为从 registry 派生；新增类型无需改核心文件 | 类型系统重构 |
| E3.2 | 第三方 Widget SDK | 提供 `createWidgetPlugin()` API + 类型定义 + 打包配置；外部可独立打包发布 widget 包 | E3.1 |
| E3.3 | Widget 市场雏形 | Widget 模板页（/widget-docs）升级为可安装的市场入口；支持从 npm 包加载 widget | E3.2 |
| E3.4 | 扩展文档 | 完整的「第三方 widget 开发指南」，含脚手架、API 参考、示例 | E3.2 |

**关键决策**：E3.1 是架构级改造，影响整个类型系统。需评估对现有 85+ widget 的迁移成本，可能需要提供兼容层。

## 四、关键决策点

| 决策 | 背景 | 选项 | 倾向 |
|------|------|------|------|
| D1 表单与大屏是否分叉 | 两头兼顾风险 | A 单一编辑器双模式 / B 拆为两个产品 | A（已投入双模式架构，复用渲染引擎，分叉成本更高） |
| D2 撤销重做方案 | JSON 深拷贝技术债 | A 增量 diff / B 结构化 patch（如 immer） / C 限制历史步数 | A（增量 diff 平衡复杂度与收益） |
| D3 容器嵌套 | 布局能力 vs 复杂度 | A 保持单层 / B 放开多层嵌套 | 先调研需求再定，倾向有限放开（最多 2 层） |
| D4 监控自研 vs 三方 | 上报通道 | A Sentry / B 自研（对接平台已有监控） / C 先 console 增强 | B（对接平台已有基础设施，避免多套） |
| D5 i18n 文案抽取 | 工作量 | A 全量一次性 / B 按模块渐进 | B（EditorView → Toolbar → PropertyPanel 逐模块） |

## 六、风险与依赖

| 风险 | 影响 | 缓解 |
|------|------|------|
| E1.2 虚拟化重写 EditorOverlay 命中计算 | 拖拽/选中回归风险 | 先做技术预研 + 影子实现，回归测试全覆盖后再切换 |
| E3.1 类型系统重构 | 影响 85+ widget 迁移 | 提供兼容层，分批迁移，保留旧类型别名 |
| 双模式复杂度翻倍 | 维护成本上升 | composable 抽象模式差异，渲染层统一 |
| 埋点接入需平台侧上报通道 | 跨项目依赖 | 早期与 server 侧对齐接口，先用 console 增强过渡 |
| 容器嵌套放开后碰撞检测性能 | 卡顿 | 限制嵌套层数（≤ 2），递归检测加缓存 |
